# 📋 版本管理规范

**创建时间**: 2026-04-18 13:00  
**适用范围**: 所有产品开发迭代  
**目的**: 确保每次迭代可追溯、可回滚、可审计

---

## 🎯 核心原则

1. **每次迭代必记录** - 无论大小变更
2. **版本信息必完整** - 时间、内容、影响范围
3. **回滚方案必明确** - 出现问题能快速恢复
4. **文档索引必清晰** - 快速找到对应版本文档

---

## 📁 文档结构

```
/workspace/
├── VERSION.md                      # 版本总览（当前版本详情）
├── CHANGES.md                      # 变更日志（详细功能说明）
├── V28.0-版本迭代日志.md            # 迭代历史（完整版本树）
├── V28.0-技术方案.md                # 技术方案（按版本）
├── V28.0-产品说明.md                # 产品文档（按版本）
└── 版本管理规范.md                  # 本文档（规范说明）
```

## 🔄 迭代流程

### 1. 迭代前准备

**检查清单**:
- [ ] 确认当前版本号（查看 VERSION.md）
- [ ] 确定本次迭代类型（功能/优化/Bug 修复）
- [ ] 预估影响范围（哪些文件会变更）
- [ ] 备份当前版本（git commit）

### 2. 迭代开发

**开发规范**:
- ✅ 小步提交（每完成一个功能点就 commit）
- ✅ 提交信息规范（带版本号）
- ✅ 代码注释清晰（特别是核心逻辑）
- ✅ 测试同步进行（开发完立即测试）

### 3. 迭代完成

**必须完成的文档**:
- [ ] 更新 `V28.0-版本迭代日志.md`
  - 添加新版本记录
  - 记录核心变更
  - 编写回滚方案
- [ ] 更新 `VERSION.md`
  - 更新版本历史表格
  - 更新当前版本信息
- [ ] 更新 `CHANGES.md`（如有新功能）
  - 详细描述功能
  - 使用说明
- [ ] 创建/更新相关技术文档
  - 技术方案
  - 产品说明
  - 使用指南

### 4. Git 提交

**提交规范**:
```bash
git add -A
git commit -m "V{版本号} - {版本描述}

🆕 核心功能:
- 功能 1
- 功能 2

🔧 技术实现:
- 实现细节 1
- 实现细节 2

📝 新增文档:
- 文档 1
- 文档 2

📊 变更统计:
- 文件 1: +X 行
- 文件 2: -Y 行

✅ 测试要点:
- 测试点 1
- 测试点 2"
```

**示例**:
```bash
git commit -m "V28.0.2 - 套餐系统版

🆕 核心功能:
- 根据会员等级决定名字数量
- 套餐选择功能
- 差异化限流

🔧 技术实现:
- AI_CONFIG.TIER_LIMITS 配置
- getNameLimitByTier() 函数
- buildNamingPrompt() 动态 Prompt

📝 新增文档:
- V28.0-版本迭代日志.md
- V28.0-套餐系统说明.md

📊 变更统计:
- index.html: +100 行
- 新增文档：2 个（11KB）

✅ 测试要点:
- 不同套餐生成不同数量名字
- 套餐状态持久化
- 限流逻辑正确"
```

### 5. 版本发布

**发布检查清单**:
- [ ] 所有文档已更新
- [ ] Git 提交已完成
- [ ] 核心功能已测试
- [ ] 回滚方案已验证
- [ ] 相关人员已通知

## 📊 版本命名规范

### 版本号格式

`V{主版本}.{次版本}.{修订版本} - {描述}`

**示例**:
- `V28.0.0` - 重大版本（新功能、架构变更）
- `V28.0.1` - 小版本（功能增强、优化）
- `V28.0.2` - 修订版（Bug 修复、小优化）

### 版本类型标识

在提交信息和文档中使用 emoji 标识：

| Emoji | 含义 | 使用场景 |
|-------|------|---------|
| 🆕 | 新功能 | 新增功能模块 |
| 🔧 | 优化 | 功能优化、性能提升 |
| 🐛 | 修复 | Bug 修复 |
| 📝 | 文档 | 文档更新 |
| 🎨 | UI | 界面优化 |
| 🚀 | 性能 | 性能优化 |
| 🚨 | 紧急 | 紧急修复 |
| ⚡ | 重构 | 代码重构 |
| 📦 | 依赖 | 依赖库更新 |

## 🚨 紧急回滚流程

### 生产环境紧急回滚

**场景**: 线上版本出现严重 Bug，需要立即恢复

**流程**:
```bash
# 1. 立即找到上一个稳定版本
git log --oneline | grep "✅"

# 2. 回滚代码
git checkout {stable-version-hash} -- index.html

# 3. 测试核心功能
# 在本地快速验证

# 4. 提交回滚
git commit -m "🚨 紧急回滚到 {stable-version}"

# 5. 推送到生产
git push origin master

# 6. 验证线上版本
# 访问生产环境确认

# 7. 通知相关人员
# 同步回滚信息
```

### 回滚后处理

- [ ] 分析 Bug 原因
- [ ] 修复问题
- [ ] 重新测试
- [ ] 再次发布新版本

**统计维度**:
- 发布版本数量
- 功能新增数量
- Bug 修复数量

2026-04 版本统计:

## 🎓 培训要点

### 新人培训

1. **版本重要性** - 为什么需要版本管理
2. **文档规范** - 如何写版本迭代日志
3. **Git 流程** - 提交、回滚、分支
4. **工具使用** - 相关文档模板

### 常见错误

❌ **错误 1**: 迭代后不更新文档
- **后果**: 无法追溯历史，回滚困难
- **解决**: 强制执行检查清单

❌ **错误 2**: 提交信息不规范
- **后果**: 无法快速定位版本
- **解决**: 使用模板，Code Review

❌ **错误 3**: 不回滚测试
- **后果**: 紧急情况下回滚失败
- **解决**: 定期演练回滚流程
